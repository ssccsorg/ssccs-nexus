# ADR-0003: Explainability in EdgeQuake Information Retrieval

**Issue**: [#128](https://github.com/raphaelmansuy/edgequake/issues/128)
**Status**: Proposed
**Decision**: Candidate A — Property-Enriched Provenance + ExplainTrace
**Date**: 2026-04-02

---

## Context

Users in regulated and high-stakes domains need to understand WHY EdgeQuake
produced a specific answer. The current system returns sources but provides
no insight into retrieval reasoning, extraction method, or answer confidence.

## Decision

### Phase 1: Build-Time Extraction Provenance

Extend entity/relationship merger to write additional properties:

```
+------------------------------------------------------------------+
|  ENTITY MERGER — NEW PROPERTIES                                  |
|                                                                  |
|  Existing:                       Added:                          |
|  - source_chunk_ids              - extraction_method              |
|  - source_document_id            - extraction_model               |
|  - source_file_path              - gleaning_iterations            |
|                                  - extraction_count               |
|                                  - source_documents (Vec)         |
|                                  - first_extracted_at             |
|                                  - last_updated_at                |
+------------------------------------------------------------------+
|  RELATIONSHIP MERGER — NEW PROPERTIES                            |
|                                                                  |
|  Existing:                       Added:                          |
|  - source_chunk_id               - extraction_method              |
|  - source_document_id            - extraction_model               |
|  - source_file_path              - extraction_weight              |
+------------------------------------------------------------------+
```

Data flows from `ExtractionMetadata` (already constructed in pipeline) through
to graph properties:

```
Pipeline::process()
  → Extractor::extract()
      → ExtractionMetadata { model, tokens, gleaning, ... }
  → Merger::merge()
      → write ExtractionMetadata fields to node/edge properties
```

### Phase 2: Query-Time ExplainTrace

Add `ExplainCollector` to the query engine that records decisions:

```
+------------------------------------------------------------------+
|  QUERY WITH EXPLAIN=TRUE                                         |
|                                                                  |
|  Query: "What caused the OEE drop?"                             |
|    |                                                             |
|    v                                                             |
|  1. Query Analysis                                               |
|     keywords: ["OEE", "drop", "cause"]                           |
|     mode: hybrid (entity names + causal theme)                   |
|     -> record in ExplainCollector                                |
|    |                                                             |
|    v                                                             |
|  2. Vector Search                                                |
|     candidates: 47, selected: 20 (min_score=0.1)                |
|     top: [chunk-5: 0.92, chunk-12: 0.88, chunk-3: 0.85]        |
|     -> record in ExplainCollector                                |
|    |                                                             |
|    v                                                             |
|  3. Graph Traversal                                              |
|     entities found: 12, edges: 28                                |
|     OEE (degree=15) → MACHINE_A (degree=8) → DEFECT_X (degree=3)|
|     -> record in ExplainCollector                                |
|    |                                                             |
|    v                                                             |
|  4. Context Assembly                                             |
|     tokens: 4200 / 30000 budget                                  |
|     chunks: 15 included, 5 truncated                             |
|     entities: 12, relationships: 8                               |
|     -> record in ExplainCollector                                |
|    |                                                             |
|    v                                                             |
|  5. Confidence Scoring                                           |
|     overall: 0.82                                                |
|     source_coverage: 0.90 (9/10 claims have sources)             |
|     avg_similarity: 0.78 (mean vector score)                     |
|     graph_connectivity: 0.75 (avg degree of cited entities)      |
|    |                                                             |
|    v                                                             |
|  Response:                                                       |
|    answer: "The OEE drop was caused by..."                       |
|    explain: { trace: [...], confidence: 0.82 }                   |
+------------------------------------------------------------------+
```

**Query request** — add `explain` field:

```json
{
  "query": "What caused the OEE drop?",
  "explain": true
```

## Consequences

### Positive

- Full answer traceability from response to source document
- Confidence score enables automated quality checks
- Opt-in: no performance penalty when explain=false
- Entity provenance endpoint enables standalone exploration
- Compatible with future upgrade to dedicated provenance table

### Negative

- Increased graph storage per entity (~250 bytes additional)
- ExplainTrace collector adds code complexity to query engine
- Confidence scoring algorithm needs tuning per use case
- Frontend explain panel adds UI complexity

---

## Anticipated Roadblocks

| Roadblock | Resolution |
|-----------|------------|
| ExtractionMetadata not available in merger | Thread through `ProcessingContext` from pipeline to merger |
| Multi-doc entity source_documents accumulation | Use `serde_json::Value::Array` — append new doc IDs on merge |
| ExplainCollector threading through query engine | Use `Arc<Mutex<ExplainCollector>>` passed to each query step |
| Confidence scoring calibration | Start with simple weighted average, tune with user feedback |
| Streaming explain event ordering | Emit after Context, before first Token |
| Apache AGE property size limits | JSON properties have no practical limit; verified in docs |

## Definition of Done

- [ ] Entity merger writes extraction_method, extraction_model, source_documents to node properties
- [ ] Relationship merger writes extraction_method, extraction_model to edge properties
- [ ] Multi-document entities accumulate ALL source_document_ids (not just last)
- [ ] Query request accepts `explain: bool` field (default false)
- [ ] When explain=true, response includes `explain` field with retrieval trace
- [ ] ExplainTrace records: query analysis, vector search, graph traversal, context assembly
- [ ] Confidence score computed from source coverage, similarity, connectivity
- [ ] `GET /entities/:name/provenance` returns extraction history per document
- [ ] When explain=false, zero performance overhead (no collector instantiated)
- [ ] Streaming: `Explain` event emitted after Context, before first Token
- [ ] All existing query tests pass (explain=false is default)
- [ ] 100% of ExplainTrace fields documented in OpenAPI schema

## Test Stories

### TS-001: Extraction provenance persisted

```
GIVEN  document "report.pdf" processed through LLM extractor with gpt-5-nano
WHEN   entity "OEE" is merged into graph
THEN   node properties include:
       extraction_method = "llm"
       extraction_model = "gpt-5-nano"
       extraction_count = 1
       source_documents = ["doc-001"]
```

### TS-002: Multi-document provenance accumulation

```
GIVEN  entity "OEE" already exists from doc-001 (extraction_count=1)
WHEN   doc-003 also contains "OEE" and is processed
THEN   entity "OEE" properties updated:
       extraction_count = 2
       source_documents = ["doc-001", "doc-003"]
       source_chunk_ids includes chunks from BOTH documents
```

### TS-003: Query with explain=false (default, no overhead)

```
GIVEN  workspace with indexed documents
WHEN   user queries { "query": "What is OEE?" }
THEN   response has answer + sources + stats
AND    response does NOT have "explain" field
AND    query latency is within normal bounds (no collector overhead)
```

### TS-004: Query with explain=true

```
GIVEN  workspace with indexed documents
WHEN   user queries { "query": "What is OEE?", "explain": true }
THEN   response includes "explain" field with:
       query_analysis.selected_mode = "hybrid"
       retrieval_steps has at least 1 step
       context_assembly.total_tokens > 0
       confidence.overall between 0.0 and 1.0
```

### TS-005: Entity provenance endpoint

```
GIVEN  entity "OEE" extracted from 2 documents
WHEN   GET /api/v1/entities/OEE/provenance?workspace_id=xxx
THEN   response includes:
       entity = "OEE"
       sources has 2 entries
       sources[0].extraction_method = "llm"
       sources[1].extraction_method = "sota"
       description_history has at least 2 entries
```

### TS-006: Confidence score reflects source quality

```
GIVEN  query with 5 high-score chunks (>0.85) from 3 different documents
AND    cited entities have average degree > 5
WHEN   explain=true query is executed
THEN   confidence.overall > 0.7
AND    confidence.source_coverage > 0.8
AND    confidence.avg_similarity > 0.8
AND    confidence.graph_connectivity > 0.6
```

### TS-007: Streaming explain event

```
GIVEN  explain=true on streaming query
WHEN   SSE events are received
THEN   event order is: Context → Explain → Token... → Done
AND    Explain event contains full ExplainTrace
```

### TS-008: SOTA extractor provenance differs from LLM

```
GIVEN  document processed by SOTA extractor
WHEN   entity merged into graph
THEN   extraction_method = "sota"
AND    extraction_model = "rule-based"
AND    gleaning_iterations = 0
```
