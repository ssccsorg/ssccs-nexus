# ADR-0004: Display Link Type (Relation) in the Graph View

**Issue**: [#91](https://github.com/raphaelmansuy/edgequake/issues/91)
**Status**: Proposed
**Decision**: Candidate A — Backend Rename + Sigma.js Styling
**Date**: 2026-04-02

---

## Context

Edge labels never display in the graph view even when "Show Edge Labels" is
enabled. Root cause: backend serializes the field as `edge_type` but frontend
expects `relationship_type`. Result: label is always `undefined`.

## Decision

### Step 1: Rename `edge_type` to `relationship_type` in API

```rust
// edgequake-api/src/handlers/graph_types.rs

#[derive(Debug, Clone, Serialize, ToSchema)]
pub struct GraphEdgeResponse {
    pub source: String,
    pub target: String,

    #[serde(rename = "relationship_type")]
    pub edge_type: String,

    pub weight: f32,

    #[serde(skip_serializing_if = "Option::is_none")]
    pub description: Option<String>,

    pub properties: serde_json::Value,
}
```

### Step 2: Populate `description` at all construction sites

4 sites produce `GraphEdgeResponse`. Each must now extract `description`:

```
graph_stream.rs:          e.properties.get("description")
graph_query/search.rs:    e.properties.get("description")
graph_query/traversal.rs: e.properties.get("description") (×2)
```

### Step 3: Fix `search.rs` property key

```rust
// BEFORE (wrong key):
.get("relationship_type")

// AFTER (correct key — matches what merger writes):
.get("relation_type")
```

### Step 4: Add Sigma.js edge label styling

```typescript
// graph-renderer.tsx, Sigma constructor options

edgeLabelSize: 11,
edgeLabelColor: {
    color: isDark ? '#9ca3af' : '#6b7280',
},
edgeLabelFont: 'Inter, ui-sans-serif, system-ui, sans-serif',
```

### Data Flow After Fix

```
+--------------------------------------------------------------------+
|  FIXED DATA FLOW                                                   |
|                                                                    |
|  1. PostgreSQL:                                                    |
|     properties = { "relation_type": "WORKS_FOR" }                  |
|                                                                    |
|  2. Backend extracts:                                              |
|     edge_type = properties.get("relation_type")  // "WORKS_FOR"    |
|                                                                    |
|  3. Backend serializes (with serde rename):                        |
|     { "relationship_type": "WORKS_FOR", ... }                      |
|       ^^^^^^^^^^^^^^^^^^                                           |
|       Matches frontend expected field name                         |
|                                                                    |
|  4. Frontend GraphEdge:                                            |
|     GraphEdge { relationship_type: "WORKS_FOR" } ✓                 |
|                                                                    |
|  5. Graph renderer:                                                |
|     graph.addEdge(..., { label: "WORKS_FOR" }) ✓                   |
|                                                                    |
|  6. Sigma.js renders label on edge ✓                               |
+--------------------------------------------------------------------+
```

---

## Consequences

### Positive

- Edge labels display correctly with relationship type text
- Labels adapt to dark/light mode via `edgeLabelColor`
- Performance unaffected (labels already disabled for very large graphs)
- search.rs property key fix prevents silent edge type bugs

### Negative

- API field rename from `edge_type` → `relationship_type` breaks external
  consumers relying on `edge_type` key name

### Mitigation

- EdgeQuake API is not yet public/versioned — internal consumers only
- Frontend already expects `relationship_type` so no frontend breakage
- If backward compat needed: add `#[serde(alias = "edge_type")]` on
  any deserialization side

---

## Anticipated Roadblocks

| Roadblock | Resolution |
|-----------|------------|
| `edge_type` used by external API consumers | API is internal; no known external consumers |
| Sigma.js edge labels overlap with edges | `edgeLabelSize: 11` + curved edges provides clearance |
| Edge labels unreadable at small zoom | Sigma.js handles label visibility via LOD by default |
| Dark mode label visibility | `edgeLabelColor` set per theme |
| Very long relationship types overflow | CSS/Sigma truncation handles gracefully |

## Definition of Done

- [ ] `GraphEdgeResponse` serializes `edge_type` as `"relationship_type"` in JSON
- [ ] `description` field added to `GraphEdgeResponse` and populated from properties
- [ ] `search.rs` reads `relation_type` (not `relationship_type`) from properties
- [ ] Edge labels render with relationship type text when "Show Edge Labels" is on
- [ ] Edge labels have appropriate size and color for both light and dark modes
- [ ] Edge labels are hidden for very large graphs (>500 nodes, existing behavior)
- [ ] No regression in graph rendering performance
- [ ] Existing graph API tests still pass
- [ ] OpenAPI schema updated (`ToSchema` auto-handles via derive)

## Test Stories

### TS-001: Edge labels display when enabled

```
GIVEN  graph with entity "Alice" connected to "Acme Corp" via "WORKS_FOR"
AND    "Show Edge Labels" toggle is ON (settings or graph controls)
WHEN   graph renders
THEN   the edge between Alice and Acme Corp displays "WORKS_FOR" label
AND    label is readable (correct size and contrast)
```

### TS-002: Edge labels hidden when disabled

```
GIVEN  same graph as TS-001
AND    "Show Edge Labels" toggle is OFF
WHEN   graph renders
THEN   NO edge labels are displayed
AND    edges still render as curved arrows
```

### TS-003: Dark mode label visibility

```
GIVEN  dark mode enabled
AND    edge labels enabled
WHEN   graph renders
THEN   edge labels are light colored (#9ca3af) against dark background
AND    labels are readable
```

### TS-004: Light mode label visibility

```
GIVEN  light mode enabled
AND    edge labels enabled
WHEN   graph renders
THEN   edge labels are dark colored (#6b7280) against light background
AND    labels are readable
```

### TS-005: Very large graph disables labels

```
GIVEN  graph with >500 nodes
AND    edge labels enabled in settings
WHEN   graph renders
THEN   edge labels are NOT rendered (performance protection)
AND    node labels follow their own LOD rules
```

### TS-006: API returns relationship_type field

```
GIVEN  graph with edge having relation_type="WORKS_FOR" in properties
WHEN   GET /api/v1/graph/stream is called
THEN   edge JSON contains "relationship_type": "WORKS_FOR"
AND    edge JSON does NOT contain "edge_type" key
```

### TS-007: search.rs returns correct edge type

```
GIVEN  edge with properties { "relation_type": "COLLABORATES_WITH" }
WHEN   graph search endpoint is called
THEN   edge_type is "COLLABORATES_WITH" (not "RELATED_TO" fallback)
```

### TS-008: Multiple edges between same nodes

```
GIVEN  "Alice" connected to "Bob" via both "KNOWS" and "WORKS_WITH"
AND    edge labels enabled
WHEN   graph renders
THEN   both labels display on their respective curved edges
AND    labels don't overlap (curved arrow separation)
```
