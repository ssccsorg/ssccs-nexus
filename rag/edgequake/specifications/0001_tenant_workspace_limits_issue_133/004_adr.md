# ADR-0001: Tenant Workspace Limits API

**Issue**: [#133](https://github.com/raphaelmansuy/edgequake/issues/133)
**Status**: Proposed
**Decision**: Candidate A — Thin API Layer
**Date**: 2026-04-02

---

## Context

EdgeQuake uses compile-time plan defaults for `max_workspaces`. Platform
operators need runtime control over tenant quotas without redeployment.

## Decision

Implement two admin-only API endpoints:

### 1. Update Tenant Quota

```
PATCH /api/v1/admin/tenants/:tenant_id/quota
```

```json
// Request
{ "max_workspaces": 750 }

// Response 200
{
  "tenant_id": "uuid",
  "max_workspaces": 750,
  "previous_max_workspaces": 500,
  "current_workspace_count": 42
}
```

### 2. Update Server-Wide Default

```
PATCH /api/v1/admin/config/defaults
```

```json
// Request
{ "default_max_workspaces": 200 }

// Response 200
{
  "default_max_workspaces": 200,
  "note": "Applies to newly created tenants only. Not retroactive."
}
```

### Architecture

```
+------------------------------------------------------------------+
|                    QUOTA UPDATE FLOW                              |
|                                                                  |
|  Admin Client                                                    |
|    |                                                             |
|    v                                                             |
|  PATCH /api/v1/admin/tenants/:id/quota                           |
|    |                                                             |
|    v                                                             |
|  +---------------------------+                                   |
|  | auth_middleware            |  Verify: role == Admin            |
|  +---------------------------+                                   |
|    |                                                             |
|    v                                                             |
|  +---------------------------+                                   |
|  | update_tenant_quota()     |                                   |
|  | handler                   |                                   |
|  +---------------------------+                                   |
|    |                                                             |
|    |  1. Fetch tenant (SELECT FOR UPDATE)                        |
|    |  2. Count current workspaces                                |
|    |  3. Validate: new_value >= current_count                    |
|    |  4. Validate: new_value > 0                                 |
|    |  5. Update metadata JSONB                                   |
|    |  6. Log change via tracing::info!                           |
|    |                                                             |
|    v                                                             |
|  +---------------------------+                                   |
|  | PostgreSQL                |                                   |
|  | UPDATE tenants            |                                   |
|  | SET metadata = jsonb_set  |                                   |
|  |   (metadata,              |                                   |
|  |    '{max_workspaces}',    |                                   |
|  |    '750')                 |                                   |
|  | WHERE tenant_id = $1      |                                   |
|  +---------------------------+                                   |
+------------------------------------------------------------------+
```

### Validation Rules

| Rule | Check                                  | Error                                        |
| ---- | -------------------------------------- | -------------------------------------------- |
| V1   | `new_value > 0`                        | 400: "max_workspaces must be positive"       |
| V2   | `new_value >= current_workspace_count` | 409: "Cannot reduce below current count (N)" |
| V3   | `new_value <= 10000`                   | 400: "max_workspaces exceeds sanity limit"   |
| V4   | Caller is Admin                        | 403: "Insufficient permissions"              |
| V5   | Tenant exists and is active            | 404: "Tenant not found"                      |

### Server-Wide Default Storage

```
Resolution order for new tenant default_max_workspaces:
  1. server_config table → key "default_max_workspaces"
  2. EDGEQUAKE_DEFAULT_MAX_WORKSPACES env var
  3. TenantPlan::default_max_workspaces() (compile-time fallback)
```

Requires one new table:

```sql
CREATE TABLE IF NOT EXISTS server_config (
  key        TEXT PRIMARY KEY,
  value      JSONB NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Concurrency Safety

```sql
-- Use SELECT FOR UPDATE to prevent TOCTOU race
BEGIN;
SELECT * FROM tenants WHERE tenant_id = $1 FOR UPDATE;
-- count workspaces
SELECT COUNT(*) FROM workspaces WHERE tenant_id = $1;
-- validate and update
UPDATE tenants SET metadata = jsonb_set(...), updated_at = NOW()
  WHERE tenant_id = $1;
COMMIT;
```

## Consequences

### Positive

- Operators can adjust quotas without redeployment
- Monotonic increase prevents orphan state
- `check_workspace_limit()` works unchanged
- No schema migration for `tenants` table

### Negative

- No built-in change history (logging only)
- `server_config` table requires migration

### Neutral

- UI can be built on top of these endpoints later

## Anticipated Roadblocks

| Roadblock                          | Resolution                                                     |
| ---------------------------------- | -------------------------------------------------------------- |
| `metadata` JSONB key casing        | Use exact key `"max_workspaces"` as in existing code           |
| `WorkspaceService` trait extension | Add `update_tenant_quota()` method                             |
| Auth middleware for admin routes   | Reuse existing `Role::Admin` check                             |
| sqlx compile-time query check      | Use `sqlx::query!` with proper types, or dynamic `sqlx::query` |

## Definition of Done

- [ ] `PATCH /api/v1/admin/tenants/:id/quota` accepts `{ "max_workspaces": N }` and persists it.
- [ ] Rejects decrease below current workspace count with 409.
- [ ] Rejects non-positive values with 400.
- [ ] Rejects non-admin callers with 403.
- [ ] `PATCH /api/v1/admin/config/defaults` updates server-wide default.
- [ ] New tenants use server-wide default (not compile-time constant).
- [ ] Existing tenants are NOT retroactively affected.
- [ ] `server_config` migration runs cleanly.
- [ ] All existing workspace creation tests still pass.
- [ ] Concurrency test: two parallel updates on same tenant yield correct state.
- [ ] Tracing log emitted for every quota change with tenant_id, old_value, new_value.

## Test Stories

### TS-001: Increase tenant quota

```
GIVEN  tenant "acme" has max_workspaces = 100 and 42 active workspaces
WHEN   admin PATCH /admin/tenants/:acme_id/quota { "max_workspaces": 200 }
THEN   response 200 with previous=100, new=200, current_count=42
AND    tenant can now create up to 200 workspaces
```

### TS-002: Reject decrease below current usage

```
GIVEN  tenant "acme" has max_workspaces = 100 and 80 active workspaces
WHEN   admin PATCH /admin/tenants/:acme_id/quota { "max_workspaces": 50 }
THEN   response 409 with message "Cannot reduce below current count (80)"
AND    max_workspaces remains 100
```

### TS-003: Reject non-positive value

```
GIVEN  any tenant
WHEN   admin PATCH /admin/tenants/:id/quota { "max_workspaces": 0 }
THEN   response 400 with message "max_workspaces must be positive"
```

### TS-004: Reject non-admin caller

```
GIVEN  caller has role "User"
WHEN   PATCH /admin/tenants/:id/quota { "max_workspaces": 200 }
THEN   response 403
```

### TS-005: Update server-wide default

```
GIVEN  server default is 500 (from TenantPlan::Pro)
WHEN   admin PATCH /admin/config/defaults { "default_max_workspaces": 200 }
THEN   next tenant created gets max_workspaces = 200
AND    existing tenants still have their original limits
```

### TS-006: Concurrent quota updates

```
GIVEN  tenant "acme" has max_workspaces = 100
WHEN   admin sends TWO concurrent PATCH with max_workspaces = 200 and 300
THEN   exactly one wins (last-write-wins within transaction)
AND    final value is either 200 or 300 (not corrupted)
```

### TS-007: Sanity cap enforcement

```
GIVEN  any tenant
WHEN   admin PATCH /admin/tenants/:id/quota { "max_workspaces": 99999 }
THEN   response 400 with message "max_workspaces exceeds sanity limit (10000)"
```
