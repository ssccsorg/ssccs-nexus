# ADR-0005: Custom Entity Configuration from UI

**Issue**: [#85](https://github.com/raphaelmansuy/edgequake/issues/85)
**Status**: Implemented
**Decision**: Candidate A — Workspace Metadata + Preset System
**Date**: 2026-04-03

---

## Context

Entity types are hardcoded in the pipeline extractors. Domain-specific
workspaces (manufacturing, healthcare, legal) cannot configure entity types,
causing low recall and wrong typing of domain entities.

## Decision

### Step 1: Backend — Add `entity_types` to `CreateWorkspaceRequest`

```rust
// edgequake-core/src/types/multitenancy/requests.rs

pub struct CreateWorkspaceRequest {
    pub name: String,
    // ...existing fields...

    /// Custom entity types for extraction pipeline.
    /// If None, uses default entity types.
    /// @implements SPEC-085: Custom entity configuration
    pub entity_types: Option<Vec<String>>,
}

impl CreateWorkspaceRequest {
    /// Set custom entity types.
    pub fn with_entity_types(mut self, types: Vec<String>) -> Self {
        self.entity_types = Some(types);
        self
    }
}
```

### Step 2: Backend — Persist to workspace metadata

```rust
// workspace_service_impl.rs — in create_workspace()

if let Some(ref entity_types) = request.entity_types {
    // Validate: uppercase, non-empty, max 50, no duplicates
    let validated: Vec<String> = entity_types
        .iter()
        .map(|t| t.trim().to_uppercase().replace(' ', "_"))
        .filter(|t| !t.is_empty())
        .collect::<Vec<_>>();
    
    let unique: Vec<String> = validated.into_iter()
        .collect::<std::collections::HashSet<_>>()
        .into_iter()
        .take(50)
        .collect();
    
    metadata.insert("entity_types".to_string(), json!(unique));
}
```

### Step 3: Backend — Read entity types in pipeline creation

```rust
// In document upload handler or pipeline factory

fn get_workspace_entity_types(workspace: &Workspace) -> Vec<String> {
    workspace.metadata
        .as_ref()
        .and_then(|m| m.get("entity_types"))
        .and_then(|v| serde_json::from_value::<Vec<String>>(v.clone()).ok())
        .unwrap_or_else(default_entity_types)
}

// When creating extractor:
let entity_types = get_workspace_entity_types(&workspace);
let extractor = SOTAExtractor::new(llm.clone())
    .with_entity_types(entity_types.clone());
// OR
let extractor = LLMExtractor::new(llm.clone())
    .with_entity_types(entity_types);
```

### Step 4: Backend — Fix LLMExtractor default inconsistency

Align `LLMExtractor::new()` defaults with `default_entity_types()`:

```rust
// Before: 7 types (missing DATE, DOCUMENT)
entity_types: vec!["PERSON", "ORGANIZATION", "LOCATION", "EVENT",
                   "CONCEPT", "TECHNOLOGY", "PRODUCT"]

// After: use shared default
entity_types: crate::prompts::default_entity_types()
```

### Step 5: Backend — Workspace detail endpoint includes entity types

The workspace detail response should surface configured entity types so the
UI can display them:

```json
{
  "id": "...",
  "name": "Manufacturing QA",
  "metadata": {
    "entity_types": ["PERSON", "ORGANIZATION", "MACHINE", "COMPONENT", "DEFECT"],
    "llm_model": "gemma3:12b"
  }
}
```

Already surfaced via JSONB metadata — no additional work needed.

### Data Flow After Implementation

```
+--------------------------------------------------------------------+
|  COMPLETE FLOW                                                     |
|                                                                    |
|  1. User selects "Manufacturing" preset (or builds custom list)    |
|     → entity_types = [PERSON, MACHINE, COMPONENT, DEFECT, ...]    |
|                                                                    |
|  2. CreateWorkspaceRequest sent to backend:                        |
|     { name: "Mfg QA", entity_types: [...], llm_model: "..." }     |
|                                                                    |
|  3. Backend validates and stores in workspace metadata:            |
|     metadata = { "entity_types": [...], "llm_config": {...} }      |
|                                                                    |
|  4. User uploads maintenance_log.pdf                               |
|                                                                    |
|  5. Upload handler reads workspace entity_types from metadata      |
|     entity_types = workspace.metadata.entity_types                 |
|     falls back to default_entity_types() if absent                 |
|                                                                    |
|  6. Extractor configured with workspace types:                     |
|     SOTAExtractor::new(llm).with_entity_types(entity_types)        |
|                                                                    |
|  7. LLM prompt includes domain types:                              |
|     "Categorize using: PERSON, MACHINE, COMPONENT, DEFECT, ..."   |
|                                                                    |
|  8. Extraction captures: MACHINE_A, BEARING_42, VIBRATION_ANOMALY |
|     → All correctly typed as MACHINE, COMPONENT, DEFECT            |
+--------------------------------------------------------------------+
```

## Consequences

### Positive

- Domain-specific entity extraction without code changes
- Presets provide instant value for 5+ industries
- Custom types enable unlimited domains
- Pipeline builder methods already exist (zero extraction engine changes)
- Backward compatible (absent entity_types → defaults)
- Workspace detail already surfaces JSONB metadata
- Full i18n coverage in UI (en, fr, zh)
- Accessibility-first UI controls (preset pressed-state, live count, labeled remove actions)
- End-to-end UI validation with interactive browser coverage

### Negative

- Entity types are set at workspace creation (no update in v1)
- Changing entity types requires new workspace + re-upload
- Preset definitions hardcoded in frontend
- No validation that types produce good extractions

### Mitigations

- v2 can add `PATCH /workspaces/:id` with entity_types update + re-extraction
- Presets can be made configurable via tenant-level config in future
- Frontend can show extraction stats by type to validate effectiveness

---

## Anticipated Roadblocks

| Roadblock | Resolution |
|-----------|------------|
| LLMExtractor defaults differ from SOTAExtractor | Align both to use `default_entity_types()` |
| Prompt bloat with 50 entity types | 50 types add ~250 tokens; acceptable vs chunk content |
| Entity type validation (format) | Normalize: uppercase, replace spaces with underscores, deduplicate |
| Existing workspaces without entity_types | `unwrap_or_else(default_entity_types)` fallback |
| Pipeline creation doesn't read workspace metadata | Upload handler already fetches workspace; just read the field |
| Frontend i18n for presets | Preset labels are translatable via existing i18n system |

## Test Stories

### TS-001: Create workspace with default entity types

```
GIVEN  user creates workspace without specifying entity_types
WHEN   CreateWorkspaceRequest { name: "Test", entity_types: None }
THEN   workspace created successfully
AND    workspace metadata does NOT contain entity_types key
AND    pipeline uses default_entity_types() for extraction
```

### TS-002: Create workspace with manufacturing preset

```
GIVEN  user selects "Manufacturing" preset in UI
WHEN   CreateWorkspaceRequest { name: "Mfg QA", entity_types: Some(["PERSON", "MACHINE", "COMPONENT", "DEFECT", ...]) }
THEN   workspace metadata contains entity_types array
AND    pipeline uses those types for extraction
```

### TS-003: Custom entity type added

```
GIVEN  user starts with General preset
AND    user adds custom type "CIRCUIT_BOARD"
WHEN   workspace is created
THEN   entity_types includes all General types + CIRCUIT_BOARD
AND    extraction prompt contains "CIRCUIT_BOARD" in type list
```

### TS-004: Entity type normalization

```
GIVEN  user enters " machine " and "Defect Type"
WHEN   workspace is created
THEN   entity_types contains "MACHINE" and "DEFECT_TYPE"
AND    duplicates are removed
```

### TS-005: Max entity types enforced

```
GIVEN  user attempts to add 25 entity types
WHEN   workspace creation is submitted
THEN   only first 20 types are stored
OR     validation error is returned
```

### TS-006: Extraction uses workspace entity types

```
GIVEN  workspace with entity_types ["MACHINE", "DEFECT", "COMPONENT"]
AND    document text: "Machine A detected a bearing defect"
WHEN   document is uploaded and processed
THEN   extracted entities include:
       MACHINE_A (type: MACHINE)
       BEARING (type: COMPONENT)
       DEFECT-001 (type: DEFECT)
```

### TS-007: Backward compatibility

```
GIVEN  existing workspace created before this feature (no entity_types in metadata)
WHEN   document is uploaded to this workspace
THEN   pipeline uses default_entity_types()
AND    extraction produces entities with standard types (PERSON, ORGANIZATION, etc.)
```

### TS-008: Workspace detail shows entity types

```
GIVEN  workspace with custom entity_types
WHEN   user navigates to workspace settings page
THEN   entity types are displayed as tags/chips
AND    each type shows its name in UPPERCASE format
```

### TS-009: LLM extractor uses same defaults as SOTA

```
GIVEN  workspace with no custom entity_types
WHEN   LLMExtractor is created
THEN   entity_types matches default_entity_types() (9 types)
AND    entity_types matches SOTAExtractor defaults exactly
```
